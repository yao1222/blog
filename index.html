HTTP介紹

Posted on 2020-12-19 Edited on 2020-12-23

1. HTTP 的起源

HTTP 最初是Tim BernersLee 1989 年在歐洲核子研究組織（CERN）所發起的。Tim BernersLee 提出了一種能讓遠隔兩地的研究者們共享知識的設想。這個設想的基本理念是：借助多文檔之間相互關聯形成的超文本（HyperText），連成可相互參閱的WWW（World Wide Web）。用於傳輸的超文本傳輸協議（HyperText Transfer Protocol），即HTTP 由此誕生。

WWW 這一名稱，是Web 瀏覽器當年用來瀏覽超文本的客戶端應用程序時的名稱。現在則用來表示這一系列的集合，也可簡稱為Web。

HTTP 本身是一個簡單的request-response，它通常運行在TCP 之上。在OSI 七層模型中，HTTP 位於最上層。它並不涉及數據包的傳輸，只是規定了client和server之間的通信格式。定了client可能發送給server什麼樣的消息以及得到什麼樣的響應。請求和響應消息的head以ASCII 碼形式給出。

2. Client/Server

  • Client端：Browser, python or javascript app, or any app that makes HTTP request.
  • Server端：HTTP Web Server, 像是 IIS, Apache Tomcat, NodeJS, Python Tornado.

3. HTTP Request

Request example
URL https://www.google.com
method GET, POST,…
Headers proxy,cookies,…
Body upload images…

4. HTTP Response

Response example
Status Code success, ok, 404
Headers cache, image/png,…
Body index.html, json file

可到瀏覽器打開「開發人員的視窗」->「網路」就能看到我們在開啟一個網頁時，送出的request和得到的response。

5. how HTTP works?

基本上，就是client端open connection到server，sever傳回要的資送後，clinent端close connection

Client == open ==> GET ====> Server
<======== close == index.html ==== Server

所以 HTTP1.0就是這樣的概念而成的，這樣的請求呈現重複開開關關的，會～很～慢～。

Client == open ==> GET 我要首頁 ====> Server
<======== close == index.html ==== Server
Client == open ==> GET 我要圖片1 ====> Server
<======== close == img1.jpg ==== Server
Client == open ==> GET ====> Server
<======== close == img2.jpg ==== Server

這樣當然不行呀～網頁東西太多需要快一點，於是進化了 HTTP1.1
open之後我就不關，最後才關。

Client == open ==> GET 我要首頁 ====-> Server
<================ index.html ==== Server
Client ==========-> GET 我要圖片1 ====-> Server
<================ img1.jpg ==== Server
Client ==========-> GET ====-> Server
<======- CLOSE == img2.jpg ==== Server
所以這一版本出現了cache, e-tag等等概念。

HTTP1.1

  • Persisted TCP connection
  • Low Latency
  • Streaming with Chunked transfer
    (這個就是你打開一張圖片結果它慢慢由上到下載出的情況)
  • Pipelining (disable by default)

HTTP/2

  • Compression
  • Multiplexing
  • Server Push
  • SPDY
  • secure by default
  • Protocol Negotiation during TLS(NPN/ALPN)

HTTP/2 over QUIC (HTTP/3)

  • Replaces TCP with QUIC (UDP with Congestion Control)
  • All HTTP/2 features

XAMPP is not meant for production use but only for development environments. XAMPP is configured to be open as possible to allow the developer anything he/she wants. For development environments, this is great but in a production environment, it could be fatal.

Here a list of missing security in XAMPP:

  • The MySQL administrator (root) has no password.

  • ProFTPD uses the password “lampp” for user “daemon”.

What is API?

Application Programming Interface，就字面上看起來可以得知它是一個程式-編碼-介面，但介面是什麼呢？實際上是扮演什麼角色？以餐廳來說它就是一位服務生(API)，把顧客(user)的單子送交給廚房(database)去判斷，以下先來看一下這個簡單的影片了解吧！
Simple explanation video

裡面以訂機票為例，旅遊網站可以讓使用者訂各家航空公司的機票，而不需進入到特定航空公司自己的網站下訂，這邊就是使用到API。

ex.2
搶演唱會票

  • 訂票網站 => API送出訂購人資料到伺服器(某台電腦上) => 回傳訂票成功or不成功

API can control access to resources

APi可以控制資源是被誰使用，怎麼使用。像是瀏覽器跳出視窗問允不允許browser存取你的位置資訊，如果接受，就代表API可以向browser要位置資訊。

API is used for communication between services

像是把google map內嵌在自己開發的程式中

Open API / Internal API / Partner API

以公開性可分為三種API

  1. Open API (aka Public API): 使用上沒有太大的限制，像是台灣也有一個平台智慧城鄉
  2. Internal API (aka Private API): 通常是公司內部在使用，串聯各種service/team之間的作用。
  3. Partner API:給有使用權的開發者去使用，通常會是付費或合作關係才能使用。

Web service API / Private API / Partner API

以protocal, architectural去分類，比較像是用API code 接受的datatype 或 commands去做一個分類。可參考這篇文章

最常見的就是Web service API，

  1. REST
  2. JSON-RPC
  3. XML-RPC
  4. SOAP

網址設計

要對資源做最基本的檢視、新增、刪除、修改動作的話，會使用 HTTP method (GET/POST/DELETE/PUT)。

HTTP method是RESTful API的一個設計規範，使用HTTP的對資源存取的動詞，來當作對資源存取的動作，最常見的是GET/POST，其他有PUT/DELETE/PATCH。PUT的定義是對整個資源做整個置換更新，PATCH則是對資源的資料做部分更新，若能遵循這些規範，就可以讓我們在開發API給前端JavaScript或手機App存取資料時，能更快更明確了解資源的存取方式有哪些。

Route

Route? 路由是什麼？

當使用者在輸入網址時，網站會針對這個url去做相對應的資料處理，根據傳入的檔案名稱(user.php)及參數(sign-up)去做反應。Laravel將判斷網址參數以及之後要做的動作，全部交給Route去判斷。

Route就像是餐廳服務生，根據客人選的餐點(網址參數)，告訴廚師(Controller)，要上什麼菜色(資料存取動作)。

顧客(user) — 服務生(Route) — 廚師 (資料處理)

Route(服務生)檔案位置在 routes/web.php 中，所有的網址(菜色)都寫在裡面，然後指定不同菜色(網址)給不同廚師(controller)處理。

What is composer?

目前在PHP是使用composer作為套件的相依性管理系統，不同套件之間不同版本相互依賴，使用套件管理系統在指定特定版本號後(e.g. laravel “5.5.*”)，可以確保目前運行的套件版本都是可以正確運作的。

在其他語言中也有類似PHP composer的套件管理系統，像是 Node.js 的 npm 或是 Python 的 pip，都是類似的套件管理工具。而目前Laravel就是使用composer作為它的套件管理工具。

所以當在進行安裝或更新的時候，會以目前套件版本號當作參考，如果有重大改版，套件系統並不會做跨版本的套件更新，確保整個套件可以在目前版本下正常運作。
